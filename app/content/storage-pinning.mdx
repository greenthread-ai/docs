GreenThread uses a tiered storage system for model weights. Where weights are stored — and how they're transferred to GPU — determines how fast a model can wake up from sleep. Storage is managed automatically by the storage agent DaemonSet running on each GPU node.

## Configuration

Storage behavior is controlled by a single field in the Model CRD:

```yaml
apiVersion: greenthread.ai/v1alpha1
kind: Model
metadata:
  name: llama-3-1-8b
  namespace: greenthread-system
spec:
  modelName: meta-llama/Llama-3.1-8B-Instruct
  staging:
    pinned: true
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `staging.pinned` | bool | `false` | Keep model weights in CPU pinned RAM for fast wake |

That's it — one boolean controls the tradeoff between wake speed and RAM usage. The storage agent handles everything else automatically.

## How it works

### Pinned (`staging.pinned: true`)

Weights are kept in CPU pinned RAM after loading to GPU. On wake, the storage agent uses **DMA** to transfer directly from RAM to GPU.

- **Transfer speed**: ~65 GB/s (RAM → GPU)
- **Wake time**: ~600ms for a 20B model
- **Tradeoff**: Consumes host RAM proportional to model size

### Not pinned (`staging.pinned: false`)

Weights live on NVMe disk. On wake, the storage agent uses **GDS** (GPU Direct Storage) if the node is configured for it, bypassing CPU entirely. If GDS is not available, it falls back to **memory-mapped copy** (mmap).

- **Transfer speed**: ~20 GB/s (disk → GPU via GDS/mmap)
- **Wake time**: ~5-6 seconds for a 20B model with GDS, longer without
- **Tradeoff**: No RAM cost, but slower wake

<Callout type="tip" title="When to pin">
Pin your most latency-sensitive models — the ones where sub-second wake matters. For models accessed less frequently, leave pinning off to conserve RAM. With GDS-capable nodes, unpinned wake is still fast (~2s for a 20B model).
</Callout>

## Multi-node model distribution

When a model pod boots on a node that doesn't have the model weights locally, the storage agent needs to fetch them. The strategy depends on network capabilities:

<Mermaid chart={`
flowchart TD
    B[Model pod boots] --> L{Weights on\nlocal node?}
    L -->|Yes| FILL[Fill GPU from local storage]
    L -->|No| R{RDMA available?}
    R -->|Yes| RDMA[RDMA transfer\nfrom peer node]
    R -->|No| HTTP[HTTP pull\nfrom peer storage agent]
    RDMA --> FILL
    HTTP --> FILL
`} />

1. **Local** — weights are already on the node's NVMe. Fastest path.
2. **RDMA** — if the network supports it, weights are transferred directly from a peer node's memory via Remote DMA. *(Coming soon — not yet measured.)*
3. **HTTP fallback** — if RDMA isn't available, the storage agent pulls the model from a peer's HTTP endpoint, writes to local NVMe, then loads via GDS/mmap.

This means you never need to re-download from HuggingFace on additional nodes — peer storage agents serve the weights directly.

## Transfer speeds

Measured on AWS EC2 with 2x NVIDIA RTX PRO 6000 Blackwell GPUs:

| Path | Speed | Used when |
|------|-------|-----------|
| CPU pinned RAM → GPU (DMA) | ~65 GB/s | `staging.pinned: true` |
| NVMe → GPU (GDS / mmap) | ~20 GB/s | `staging.pinned: false` |
| RDMA from peer node | TBD | Model not on local node, RDMA-capable network |
| HTTP from peer node | Network-limited | Model not on local node, fallback path |

## NVMe storage

GreenThread stores converted model weights on local NVMe storage. The path is configured via the Helm chart:

```yaml
# Helm values
storage:
  modelsPath: /mnt/models      # Host path for model storage
  pinnedMemoryMB: 4096          # Total pinned memory pool size (MB)
  pinnedChunks: 8               # Number of pinned memory chunks
```

| Value | Default | Description |
|-------|---------|-------------|
| `modelsPath` | `/mnt/models` | Host path where converted models are stored on NVMe |
| `pinnedMemoryMB` | `4096` | Total CPU pinned memory pool available for staging (MB) |
| `pinnedChunks` | `8` | Number of pinned memory chunks for concurrent transfers |

The `pinnedMemoryMB` value sets the total CPU pinned memory pool. If you plan to pin multiple models, ensure this value is large enough to hold all of them. For example, 4 models at ~16GB each requires `pinnedMemoryMB: 65536`.

## Storage agent

The storage agent runs as a DaemonSet on each GPU node. It manages:

- **Model registry** — tracks converted models on local NVMe
- **CPU memory pool** — allocates and manages pinned host memory for staged models
- **Fill operations** — copies weights from the optimal source to GPU memory via IPC handles
- **Tier management** — promotes/evicts models between disk and RAM based on pinning config
- **Peer distribution** — serves model weights to storage agents on other nodes via HTTP, and via RDMA when available

Check storage agent health:

```bash
kubectl get pods -n greenthread-system -l app=gthread-storage
kubectl logs -n greenthread-system -l app=gthread-storage --tail=20
```

<Callout type="warning" title="Hugepages required">
The storage agent uses hugepages for pinned memory allocation. If the storage pod is stuck in `Pending` with `Insufficient hugepages-2Mi`, see the bootstrap script in your platform setup guide (e.g. [AWS EKS](/k8s-aws-eks)) for hugepages configuration.
</Callout>

## Planning guide

| Scenario | Config | Wake time | RAM cost |
|----------|--------|-----------|----------|
| Always-hot production model | `pinned: true` | ~600ms | Model size |
| Latency-tolerant, GDS node | `pinned: false` | ~2s | None |
| Latency-tolerant, no GDS | `pinned: false` | ~5-6s | None |
| First boot on new node (HTTP pull) | — | Network-dependent + fill | None |

For production clusters, pin your critical models and ensure `pinnedMemoryMB` covers their total weight size. Leave infrequently-used models unpinned — with GDS, unpinned wake is still under 2 seconds for a 20B model.

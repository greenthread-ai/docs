GreenThread on Kubernetes runs as a set of custom resources, controllers, and sidecars managed by the GreenThread Operator. Models are declared as `Model` CRDs and the platform handles GPU scheduling, sleep/wake lifecycle, gateway routing, and monitoring automatically.

## Components

### Controller (Operator)

The GreenThread controller watches `Model` and `GPU` custom resources and reconciles the desired state:

- Creates conversion Jobs to download models from HuggingFace and convert them to the `gthread` load format on NVMe storage
- Creates model Pods with the vLLM inference engine and sidecar containers
- Creates per-model Services and HTTPRoutes for gateway-based routing
- Manages GPU scheduling, assigning models to GPUs based on available memory
- Runs as a Deployment with leader election (2 replicas by default)

### Sidecar

Every model Pod contains a sidecar container that sits between the gateway and vLLM. The sidecar manages:

- **Wake-on-request** — If the model is sleeping, the sidecar triggers a wake (GPU memory restore from NVMe) before proxying the request
- **Sleep lifecycle** — Monitors idle timeout and initiates sleep (GPU memory checkpoint to NVMe) when the queue is empty
- **Request queuing** — Tracks in-flight requests and applies a barrier during sleep transitions to drain gracefully
- **Health and readiness probes** — Reports boot status and serving readiness to Kubernetes
- **Metrics** — Exposes Prometheus metrics for wake/sleep durations, request counts, queue depth, and GPU memory

### Storage Agent

The storage agent runs as a DaemonSet on every GPU node. It provides:

- **gRPC API** for model fill (NVMe to GPU) and evict (GPU to NVMe) operations
- **Pinned memory pool** (hugepages-backed) for high-throughput DMA transfers to GPU
- **Peer-to-peer model distribution** — Nodes discover peers via DNS and transfer models over HTTP, so conversion only runs once across the cluster
- **Model format management** — Stores models in the `gthread` format optimized for fast GPU loading

### API Server

The API server provides the GreenThread web UI and management APIs:

- Web-based dashboard for model, GPU, and node management
- REST APIs for model CRUD, GPU status, pod inspection, and cluster events
- Server-Sent Events (SSE) for real-time state updates
- Authentication via API keys (Bearer tokens)

### Gateway

GreenThread uses the Kubernetes Gateway API with Envoy Gateway as the data plane:

- A single `Gateway` resource is created by the Helm chart
- The controller creates per-model `HTTPRoute` resources automatically
- Each model gets a URL path prefix: `/<model-name>/v1/*`
- The HTTPRoute rewrites the path prefix before forwarding to the model's sidecar Service
- A cloud load balancer (NLB on AWS) is provisioned automatically via the load balancer controller

## Sidecar endpoints

Each model Pod exposes the following endpoints through its sidecar. In production, these are accessed via the gateway at `http://<gateway>/<model-name>/...`.

### Status and health

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/health` | GET | Liveness probe. Returns `503` during boot, `200` once ready. |
| `/ready` | GET | Readiness probe. Returns `200` only when the model is actively serving. |
| `/status` | GET | Model status JSON: state, boot readiness, queue depth, barrier status. |
| `/v1/models` | GET | Proxied to vLLM. Lists loaded models (works even when sleeping). |

## Custom resources

### Model CRD

The `Model` custom resource defines a model to serve. Key fields:

```yaml
apiVersion: greenthread.ai/v1alpha1
kind: Model
metadata:
  name: my-model
  namespace: greenthread-system
spec:
  modelName: meta-llama/Llama-3.1-8B-Instruct  # HuggingFace model ID
  tensorParallelSize: 1                          # GPUs per replica (power of 2)
  replicas: 1                                    # Number of serving pods
  dtype: bfloat16                                # Weight data type
  quantization: ""                               # Optional: mxfp4, awq, gptq
  gpuMemoryUtilization: "0.90"                   # Fraction of GPU VRAM for vLLM
  extraArgs:                                     # Additional vLLM CLI arguments
    - "--enforce-eager"
  sleep:
    idleTimeout: 5m                              # Sleep after 5 minutes idle
    drainTimeout: 30s                            # Max drain wait before force sleep
  fairness:
    minRuntime: 30s                              # Minimum serve time before preemption
    popular: false                               # If true, never preempted
  staging:
    pinned: false                                # Keep in CPU cache when staged
  lora:
    enabled: false                               # Enable LoRA adapter loading
  kvCache:
    enabled: false                               # Enable KV cache persistence
```

## Request flow

When a request hits the gateway:

1. **Envoy** matches the path prefix (`/<model>/v1/...`) and routes to the model's Service
2. The **sidecar** receives the request and checks the model state
3. If **serving**: the request is proxied directly to vLLM with no overhead
4. If **sleeping**: the sidecar triggers a wake via the storage agent, waits for GPU memory restore, then proxies the request
5. If the **GPU is full**: the controller's fairness policy preempts the least-recently-used model to free GPU memory, then the wake proceeds
6. **vLLM** runs inference and streams the response back through the sidecar

The entire wake-on-request flow is transparent to the client — they see slightly higher latency on the first request but the response format is identical.
